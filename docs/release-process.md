# Alfred release process

**Status:** authored 2026-07-12 (open-beta loop, Stage A). The pipeline described here is the
target; Stage D implements CI, Stage E implements the release lane. Until then this is the
checklist Martin executes manually.

## Channels

- **dev** — every merge to main; no artifacts published.
- **beta** — tagged `v0.x.0-beta.N`; GitHub pre-release; updater channel "beta".
- **stable** — tagged `v0.x.0`; GitHub release; updater channel "stable". (Post-1.0 concern.)

## Pipeline (tag → release)

1. **Tag** `v*` on a commit where CI is green (CI = the Stage-D gate: typecheck, Vitest, cargo
   tests, exclusion engine, zero-Soapbox check, gitleaks, OSV-Scanner, cargo-audit, SBOM+Grype,
   both builds, axe/Lighthouse, lychee; SHA-pinned actions; **no Trivy** — excluded per
   CVE-2026-33634).
2. **Build** (Windows x64): stage the goose sidecar **pinned by version + SHA-256**
   (stage-goose-sidecar.mjs, E4) → `npm run tauri build` → `.exe` + NSIS/MSI.
3. **Sign** (E3): Azure Trusted Signing via Entra service principal, wired behind a
   secrets-present guard — the pipeline must work unsigned until the cert exists; a release built
   unsigned states it loudly in the release notes (SmartScreen/AV reality documented in the
   install guide).
4. **SBOM + scan**: Syft SBOM (app + sidecar recorded) → Grype + OSV-Scanner; block on
   high/critical with a documented allowlist.
5. **Updater artifacts**: tauri-plugin-updater bundles + **signed latest.json** (updater keypair:
   generated by Martin, private key + password in GitHub Encrypted Secrets + offline escrow —
   password manager AND an offline copy. **Losing this key means installed users can never
   update again**; treat escrow as a release blocker).
6. **Publish**: draft GitHub Release with artifacts + CHANGELOG section (Conventional Commits →
   generated); Martin reviews and publishes. Publishing is always a human action.
7. **Post-release**: smoke-install on a clean Windows machine/VM; updater dry-check from the
   previous version against the new latest.json.

## Real-goose release gate

**Update (Stage D):** `.github/workflows/release.yml` now downloads the pinned goose 1.41.0 CLI on
the Windows runner, stages it, and runs `npm run test` before building — so the live-goose trio
**does execute in the release lane** (a Windows runner has what Linux CI cannot). `ci.yml` (Linux,
every push) still cannot, and says so in its job summary. The manual checklist below remains the
belt-and-suspenders gate for a human-verified release and for the installed-app journeys.

The three live-goose tests (permission-startup, acp-handshake, recipes.live) skip off-Windows/
without the sidecar — hosted **Linux** CI can NEVER prove goose behavior. Options:
- **Preferred:** a self-hosted Windows runner (Martin's machine or dedicated) registered for a
  `release-gate` job that stages the pinned sidecar and runs the trio + `npm run verify:all`.
- **Until that exists (the operative gate):** a manual checklist Martin executes per release, on
  Windows, and records in the release notes:
  1. `git checkout <tag>` clean tree; `npm ci`; `npm run stage:goose` (pinned version verified).
  2. `npm run verify:all` — full skip report attached; the trio must EXECUTE (not skip).
  3. Walk journeys J1–J6 (LOOP.md) on the **installed** app from the candidate installer.
  4. Byte-scan telemetry canary green (`cargo test` telemetry_tests).
A release claim about goose behavior without step 2's trio executing is invalid.

## Rollback

- **Installer:** keep the last-known-good installer attached to its release permanently; the
  install guide links "previous version".
- **Updater downgrade:** latest.json rollback = re-point to the previous artifacts; the tauri
  updater's version comparator refuses same/lower versions by default — a forced downgrade ships
  as a higher version number with the old code (documented, deliberate).
- **Data:** vault schema is versioned (G5); every schema migration ships with a down-migration
  note in docs/ before the release that introduces it. No migration, no schema change.

## Definition of Done — BETA
The LOOP.md "BETA DEFINITION OF DONE" checklist, every line with evidence class EXECUTED or
VERIFIED-LIVE (REPORTED is not release evidence).

## Definition of Done — 1.0 (draft, expand before stable)
Everything in BETA plus: signed installer (no unsigned option); self-hosted release-gate runner
(no manual-checklist substitute); crash/error reporting decision made (explicitly opt-in if it
exists at all); accessibility gate at WCAG 2.1 AA on desktop surfaces; i18n baseline (Phase 7);
upstream security-review cadence demonstrated for 3 consecutive months; threat model reviewed
against a full quarter of beta feedback.

## Tag ↔ numeric version mapping (F19, canon)

MSI rejects semver pre-release suffixes, so the tag string and the numeric app
version are distinct and BOTH required:

- **Each beta cut bumps the numeric patch** across `package.json`,
  `src-tauri/tauri.conf.json`, and `src-tauri/Cargo.toml` (+lockfiles).
- **The updater orders on the numeric** version in `latest.json`; the tag
  string is display only. An update is offered iff the feed's numeric version
  is greater than the installed one — re-cutting a tag without bumping the
  numeric means installed apps will never see the new build.
- Recorded pairs: `v0.1.0-beta.1` ↔ `0.1.0` (never published) ·
  `v0.1.0-beta.2` ↔ `0.1.1` (published 2026-07-19) · `v0.1.0-beta.3` ↔ `0.1.2`.
- Tester-facing release notes live at `docs/beta/RELEASE-NOTES-<tag>.md`; the
  release workflow reads that file into the release body and the update
  dialog's notes. Write it before pushing the tag.
