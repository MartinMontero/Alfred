# Alfred — Source QA: the three research reports

Date 22 June 2026. This is the QA pass that should have happened before the reports were used as build inputs. It records what was verified, against what, and how the findings change the Alfred brief.

## Scope (be honest about it)

This is a **targeted** audit, not an exhaustive citation-by-citation check of all ~100 references. Priority went to claims that become **hard rules** in Alfred (the AT Protocol pack above all, since those constraints get encoded as `.mdc`-style rules and CI gates), plus cross-referencing the reports' central technical claims against the primary-source research already done for the build (MCP, goose/AAIF, Tauri, Nostr libraries, Onyx). Motivational/illustrative claims were assessed for sourcing quality, not independently re-verified. Primary sources used: `atproto.com/specs/{repository,data-model,did}`, the AT Repo IETF draft, the indigo/arroba/atproto-crates implementations, the MCP spec + SDK, `aaif-goose/goose`.

## Specific findings (the ones that matter for the build)

1. **MST layer/depth algorithm — VERIFIED CORRECT.** Report: "count leading binary zeros, divide by two, round down." Canonical `atproto.com/specs/repository`: identical (SHA‑256, 2‑bit chunks, fanout 4, ÷2 round-down). My initial suspicion that it should be ÷4 was **wrong**. *However:* the indigo Go `mst` package documents this as "fanout value of **16**," which contradicts the spec's fanout 4 — a known ecosystem ambiguity. The pack encodes ÷2 (spec-correct) **and** flags the 4-vs-16 description trap so a builder reading reference code isn't misled.
2. **Encoding terminology — CURRENCY DELTA (not an error).** Report says "DAG-CBOR" throughout. The canonical spec now calls the encoding **DRISL** ("successor to DAG-CBOR"), same multicodec `0x71`, still "also known as dag-cbor," and implementations (indigo) still use DAG-CBOR method names. Same bytes, moved name. Pack: use "DRISL (a.k.a. DAG-CBOR, `0x71`)" and derive details from the live spec.
3. **Repository binary format — target v3.** Current canonical format is **v3** (atproto.com, indigo, PDS-planning sources); a mirror the report's lineage may have drawn on still says v2. Pack must target v3 specifics: `prev` field present-but-virtually-always-null, `prevData` for inductive sync, `rev` as a TID, `sig` raw bytes over the unsigned commit serialized as DRISL.
4. **Public-key encoding — VERIFY, don't trust verbatim.** Report's `publicKeyMultibase` base58btc / no-multicodec-prefix claim was not fully confirmed in this pass; key handling has evolved (K‑256 / P‑256 / P‑384). Derive from `atproto.com/specs/did` before encoding as a rule.
5. **Dependency tables & repo homes — re-derive at build time.** The report's go.mod pins, language percentages, and `bluesky-social/*` URLs are point-in-time; the ecosystem is actively moving (indigo mirrored to other hosts; a parallel `atproto-crates` set on Tangled). Generate the pack's dependency map from **live** repos + screen it through the exclusion engine; don't encode the report's tables verbatim.
6. **AAIF / governance framing — consistent, date unpinned.** The reports' "MCP + goose + AGENTS.md under the Linux Foundation's AAIF" framing matches what the build research independently found (goose now lives at `aaif-goose`). The exact founding date (cited as 9 Dec 2025) wasn't independently pinned; it's contextual, not a code rule.

## Disposition

The reports are **good enough to drive Alfred's architecture and direction** — the compass report is genuinely strong, the framework's patterns are sound, and the atproto technical core is accurate (better than typical for an LLM-generated research artifact). But all three are **secondary sources**. The correction, now baked into the brief, is a hard rule: **every constraint that becomes a CI gate, an `.mdc` rule, or a scaffold must be derived from its cited primary source at build time** — `atproto.com/specs` for the atproto pack, the MCP spec for the server, official docs elsewhere — with the reports as the map, not the territory. The framework report's empirical/marketing claims (the "study," Blue Origin, Alibaba) are not to be restated as fact in anything a builder sees.

- **READ FIRST** now labels the three reports *secondary sources / direction* and requires primary-source verification of every hard constraint before it is encoded, pointing here.
- **Phase 6 (AT Protocol pack)** now carries explicit flags: derive from the live spec; target **v3**; encoding is **DRISL (a.k.a. DAG-CBOR `0x71`)**; the layer calc is **÷2 (confirmed)** with the **fanout 4-vs-16** ambiguity noted; re-derive dependency tables and repo homes from live sources; verify key encoding against `/specs/did`.
